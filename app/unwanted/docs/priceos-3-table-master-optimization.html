Consolidation Strategy

By merging Proposals into Calendar and Events into Timeline, we eliminate expensive JOINS. This allows the LLM Agents to fetch a "Complete State" in a single query, drastically increasing answer speed and accuracy.

Table 1: listings (The Profile)

Consolidates static property data and global pricing boundaries.

Logical Group Source Tables Optimization Key
Identity listings Keeps all 15 columns (name, city, area, etc.)
Safety Rails proposals (price_floor/ceiling) Moved from proposals → listings. These are property constraints, not daily ones.
Operations New Added settings (jsonb) to store ad-hoc property configs (e.g. min/max nights).

Table 2: inventory_master (The Action State)

A per-day, per-property combined view of State + Recommendation.

Column Name Formerly In Purpose / Description
status calendar_days available / booked / blocked
current_price calendar_days The price currently live on the channel.
proposed_price proposals The price AI is recommending for this date.
reasoning proposals Full text of why the recommendation exists.
risk_level proposals Security tier for the price change (low/medium/high).
restrictions calendar_days Merged min_stay / max_stay into a JSONB for easier expansion.

Table 3: activity_timeline (The Global Context)

Chronological stream of all internal and external activity.

CREATE TABLE activity_timeline (
    id SERIAL PRIMARY KEY,
    type TEXT, -- 'reservation', 'market_event', 'holiday', 'system_log'
    start_date DATE,
    end_date DATE,
    listing_id INT REFERENCES listings(id), -- Nullable for global events
    title TEXT,
    impact_score INT, -- 0 to 100
    financials JSONB, -- Stores guest_name, payout, cleaning_fee, commission
    market_data JSONB -- Stores event_description, competitor_rates, confidence
);
Consolidated Field Mapping Logic
Title reservations.guest_name OR events.name
Financials Dumps all 21 reservation columns into a highly indexed JSONB.
Context Dumps all event_signals columns (impact, location).

No Column Information Lost

Every single column from the original 5 tables exists in this new 3-table format. By using JSONB for the "deep" details of a booking or an event, we keep the main table structure "clean" while allowing Agents to still query specific keys like financials->'payout' or market_data->'confidence'.
